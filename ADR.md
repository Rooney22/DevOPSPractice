## 1. Введение


Данный документ описывает ключевое архитектурное решение для проекта по предоставлению студентам удалённого рабочего места для выполнения и сдачи лабораторных работ.


---


## 2. Контекст

При сдаче и выполнении лабораторных работ возникают следующие проблемы:

* Отсутствие личного переносного оборудования у студентов для сдачи и выполнения лабораторных работ в университете;
* Сдача лабораторных работ через репозитории на github неудобная и не позволяет увидеть действие кода на практике;
* Нет возможности просматривать все лабораторные работы в одном месте.

В рамках учебного процесса студентам необходимо предоставить унифицированную среду разработки для выполнения лабораторных работ.


Основные требования:


* Возможность доступа к единому преднастроенноему окружению с любого устройства
* Поддержка различных языков программирования
* Возможность централизованного администрирования
* Масштабируемость под нагрузку (пиковая активность во время дедлайнов)
* Безопасность и изоляция рабочих сред (студенты не могут просматривать чужие лабораторные работы)


Сейчас рассматривается вариант хостинга облачной версии Visual Studio Code (например, на базе code-server или аналогичного решения), предоставляя каждому студенту доступ к персональному рабочему пространству через браузер.


---


## 3. Решения


### 3.1 Платформа удалённой среды разработки


Выбранное решение:


* Хостинг браузерной версии Visual Studio Code (через open-source реализацию code-server или аналогичный механизм) на серверной инфраструктуре учебного заведения или в облаке.


Описание архитектуры:


* Каждый студент получает изолированную среду (контейнер).
* Доступ осуществляется через браузер (HTTPS).
* Окружение преднастроено: компиляторы, интерпретаторы, библиотеки.
* Хранение пользовательских данных осуществляется в персональном workspace.


Преимущества:


* Отсутствие установки: студенту нужен только браузер.
* Единое окружение: исключаются проблемы «у меня не работает».
* Централизованное обновление: обновление среды производится администраторами.
* Масштабируемость: можно добавлять вычислительные ресурсы при росте нагрузки.
* Кроссплатформенность: работает на Windows, macOS, Linux, ChromeOS.


Недостатки:


* Высокая нагрузка на сервер: требуется достаточное количество CPU и RAM.
* Зависимость от стабильности сети.
* Необходимость администрирования инфраструктуры.
* Потенциальные риски безопасности при неправильной настройке изоляции.


Сравнение с альтернативными решениями:


* Локальная установка IDE:


  * Не требует серверных мощностей.
    − Отсутствие унификации среды.
    − Сложности у студентов при установке.
    − Необходимость в оборудовании у студентов.
    − Весь контроль ложится на студента.


* Удалённый доступ по RDP/VNC к виртуальным машинам:


  * Полноценная среда ОС.
    − Более высокая нагрузка на сервер.
    − Менее удобный пользовательский опыт по сравнению с веб-IDE.
    − Плохо масштабируется.


* Использование SaaS-решений (например, GitHub Codespaces):


  * Минимальное администрирование.
    − Платное решение неактуально в рамках ВУЗа.
    − Зависимость от внешнего поставщика.
    − Стоимость при масштабировании.
    − Возможные ограничения по региону или политике доступа.


* JupyterHub:


    * Удобная интеграция с .ipynb-тетрадками
    * Развертывание проще чем Visual Studio Code
    − Заточен под язык Python, поэтому актуален только для предметов на Python.

### 3.2 Изоляция пользовательских сред


Выбранное решение:


* Использование контейнеризации (Docker) для изоляции каждого студента.


Преимущества:


* Безопасность: процессы студентов изолированы друг от друга.
* Повторяемость среды: контейнеры разворачиваются из единого образа.
* Гибкость конфигурации: разные образы под разные дисциплины.
* Быстрое развертывание.


Недостатки:


* Требуются ресурсы на поддержку контейнерной инфраструктуры.
* Необходимость мониторинга и оркестрации.


Сравнение с альтернативами:


* Полноценные виртуальные машины: выше изоляция, но выше нагрузка.
* Общий сервер без изоляции: проще, но небезопасно и нестабильно.


---


### 3.3 Оркестрация и масштабирование


Выбранное решение:


* Использование Kubernetes или аналогичной системы оркестрации контейнеров.


Преимущества:


* Автоматическое масштабирование.
* Управление нагрузкой.
* Самовосстановление контейнеров.
* Гибкость конфигурации.


Недостатки:


* Сложность внедрения и поддержки.
* Порог входа для администраторов.


Сравнение с альтернативами:


* Docker Compose: проще в настройке, но плохо масштабируется.
* Ручное управление контейнерами: подходит только для небольшого числа пользователей.


---


## 4. Резюме


Для обеспечения студентов удалённым рабочим местом принято решение использовать хостинг браузерной версии Visual Studio Code с контейнерной изоляцией пользователей.


Архитектура предполагает:


* Развёртывание VS Code на серверной инфраструктуре
* Изоляцию пользователей через Docker
* Оркестрацию через Kubernetes


Данное решение обеспечивает унифицированную среду разработки, масштабируемость, безопасность и удобство использования при сохранении контроля над инфраструктурой.
