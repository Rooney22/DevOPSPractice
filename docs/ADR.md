## 1. Введение


Данный документ описывает ключевое архитектурное решение для проекта по предоставлению студентам удалённого рабочего места для выполнения и сдачи лабораторных работ.


---


## 2. Контекст

При сдаче и выполнении лабораторных работ возникают следующие проблемы:

* Отсутствие личного переносного оборудования у студентов для сдачи и выполнения лабораторных работ в университете;
* Сдача лабораторных работ через репозитории на github неудобная и не позволяет увидеть действие кода на практике;
* Нет возможности просматривать все лабораторные работы в одном месте.

В рамках учебного процесса студентам необходимо предоставить унифицированную среду разработки для выполнения лабораторных работ.


Основные требования:


* Возможность доступа к единому преднастроенному окружению с любого устройства
* Поддержка различных языков программирования
* Возможность централизованного администрирования
* Масштабируемость под нагрузку (пиковая активность во время дедлайнов)
* Безопасность и изоляция рабочих сред (студенты не могут просматривать чужие лабораторные работы)


Рассматривается использование платформы Coder для предоставления студентам удалённого рабочего места: Coder разворачивается в инфраструктуре вуза и предоставляет изолированные рабочие пространства, к которым студенты подключаются через браузер или локальный IDE (например, Visual Studio Code) по защищенному каналу.


---


## 3. Решения


## 3.1 Платформа удалённой среды разработки

Выбранное решение:

* Использование платформы Coder для хостинга удалённых сред разработки в инфраструктуре учебного заведения или в облаке.

Описание архитектуры:

* Каждый студент получает изолированную среду (контейнер), управляемую через Coder.

* Доступ осуществляется через браузер (HTTPS) или через локальный Visual Studio Code с использованием расширения удалённого подключения к workspace в Coder.

* Окружение преднастроенно: компиляторы, интерпретаторы, библиотеки, необходимые для конкретной дисциплины.

* Хранение пользовательских данных осуществляется в персональном workspace в инфраструктуре вуза.

Преимущества:

* Отсутствие установки на стороне студента: требуется только браузер или локальный IDE с расширением.

* Единое окружение: исключаются проблемы «у меня не работает» за счёт типовых образов сред.

* Централизованное администрирование и обновление окружений: обновления и изменения выполняются администраторами через Coder, возможна описания окружений как инфраструктуры‑как‑код.

* Масштабируемость: поддержка контейнерной оркестрации (например, Kubernetes) позволяет добавлять вычислительные ресурсы при росте нагрузки.

* Безопасность: код и данные остаются в инфраструктуре вуза, доступ к средам организован через тонкий клиент (браузер/IDE), настроена изоляция контейнеров.

Недостатки:

* Высокая нагрузка на сервер: требуется достаточное количество CPU и RAM под контейнеры студентов.

* Зависимость от стабильности сети для доступа к рабочему месту.

* Необходимость администрирования инфраструктуры Coder и контейнерной платформы.

Сравнение с альтернативными решениями:

* Локальная установка IDE. Не требует серверных мощностей. Отсутствие унификации среды, сложности при установке, необходимость наличия оборудования у студентов, отсутствие централизованного контроля.

* Удалённый доступ по RDP/VNC к виртуальным машинам. Полноценная среда ОС. Более высокая нагрузка на сервер, менее удобный пользовательский опыт по сравнению с веб‑IDE и тонким клиентом, хуже масштабируется.

* Использование SaaS‑решений (например, GitHub Codespaces). Минимальное администрирование. Платное решение неактуально в рамках вуза, зависимость от внешнего поставщика и ограничений по региону/политике доступа.

* JupyterHub. Удобная интеграция с .ipynb‑тетрадками, развёртывание проще для Python‑курсов. Заточен под Python и актуален только для соответствующих дисциплин.

* Visual Studio Code Server. Предоставляет браузерный доступ к знакомому интерфейсу VS Code, позволяет вынести сборки и тесты на сервер и обеспечить единообразное окружение для разработчиков. Фокусируется на запуске одного редактора и базовой удалённой разработке; отсутствует полноценный слой управления рабочими пространствами, политиками и пользователями. Для учебного сценария это усложняет централизованный контроль за курсами, конфигурациями сред и доступами. Coder, напротив, предоставляет платформу рабочих пространств с шаблонами, ролями и интеграцией с инфраструктурой вуза.

### 3.2 Изоляция пользовательских сред


Выбранное решение:


* Использование контейнеризации (Docker) для изоляции каждого студента.


Преимущества:


* Безопасность: процессы студентов изолированы друг от друга.
* Повторяемость среды: контейнеры разворачиваются из единого образа.
* Гибкость конфигурации: разные образы под разные дисциплины.
* Быстрое развертывание.


Недостатки:


* Требуются ресурсы на поддержку контейнерной инфраструктуры.
* Необходимость мониторинга и оркестрации.


Сравнение с альтернативами:


* Полноценные виртуальные машины: выше изоляция, но выше нагрузка.
* Общий сервер без изоляции: проще, но небезопасно и нестабильно.


---


### 3.3 Оркестрация и масштабирование


Выбранное решение:


* Использование Kubernetes или аналогичной системы оркестрации контейнеров.


Преимущества:


* Автоматическое масштабирование.
* Управление нагрузкой.
* Самовосстановление контейнеров.
* Гибкость конфигурации.


Недостатки:


* Сложность внедрения и поддержки.
* Порог входа для администраторов.


Сравнение с альтернативами:


* Docker Compose: проще в настройке, но плохо масштабируется.
* Ручное управление контейнерами: подходит только для небольшого числа пользователей.


---


## 4. Резюме


Для обеспечения студентов удалённым рабочим местом принято решение использовать платформу Coder для хостинга удалённых сред разработки с контейнерной изоляцией пользователей.

Архитектура предполагает:

Развёртывание Coder на серверной инфраструктуре учебного заведения (с интеграцией с Docker и Kubernetes)

Изоляцию пользователей через контейнеры (Docker)

Оркестрацию контейнеров через Kubernetes

Доступ студентов к рабочим пространствам через браузер или локальный Visual Studio Code.

Данное решение обеспечивает унифицированную среду разработки, масштабируемость, безопасность и удобство использования при сохранении контроля над инфраструктурой.
